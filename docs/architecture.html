Filter Data Flow — End to End

1
User changes a filter dropdown FilterBar.vue has v-model bindings to selectedPeriod, selectedLocation, selectedCategory, selectedStatus in useFilters.js
2
Vue watcher fires in the active view Each view has watch([selectedPeriod, selectedLocation, selectedCategory, selectedStatus], loadData) — any change triggers a reload
3
Filter state serialised to query params getCurrentFilters() returns { warehouse, category, status, month }; 'all' values are omitted from the URL
4
Axios sends HTTP GET to FastAPI Example: GET /api/orders?warehouse=London&status=Shipped&month=Q1-2025
5
Backend applies filters to in-memory data apply_filters() handles warehouse/category/status (case-insensitive). filter_by_month() expands quarters to individual months and matches order dates.
6
Pydantic validates and serialises response FastAPI validates the filtered list against the relevant Pydantic model (e.g. List[Order]) before returning JSON
7
Raw data stored in Vue refs Response assigned to allOrders.value / inventoryItems.value etc. — these are the single source of truth
8
Computed properties derive display data e.g. statusData groups orders by status; charts re-derive automatically when the ref changes — no manual re-render needed
9
Vue re-renders affected DOM nodes Templates bound to computed properties update reactively; only changed elements are patched in the virtual DOM

API Endpoints

Method Path Filters Returns
GET /api/inventory warehousecategory List of inventory items with SKU, qty, cost, reorder point, location
GET /api/inventory/{id} — Single InventoryItem
GET /api/orders warehousecategorystatusmonth List of orders with items array, status, dates, total value
GET /api/orders/{id} — Single Order
GET /api/dashboard/summary warehousecategorystatusmonth KPI totals: inventory value, low stock, pending orders, backlog count, orders value
GET /api/demand — Demand forecasts with trend (increasing / stable / decreasing)
GET /api/backlog — Backlog/shortage items with priority and days delayed
GET /api/spending/summary — Procurement, operational, labor, overhead totals + YoY change %
GET /api/spending/monthly — Monthly cost breakdown by category (stacked bar data)
GET /api/spending/categories — Spending grouped by product category
GET /api/spending/transactions — Recent transaction list
GET /api/reports/quarterly — Quarterly stats: orders, revenue, fulfillment rate
GET /api/reports/monthly-trends — Monthly order count, revenue, delivered count

Data Files — server/data/

inventory.json

Stock items: SKU, name, category, warehouse, qty on hand, reorder point, unit cost, location

orders.json

Orders with nested items array, status, warehouse, dates, total value. Statuses: Delivered · Shipped · Processing · Backordered

demand_forecasts.json

Current vs forecasted demand per SKU. Trend: increasing · stable · decreasing. Period: next 30 days

backlog_items.json

Shortage records: qty needed vs available, days delayed, priority (high / medium / low)

spending.json

Nested: spending_summary, monthly_spending (12 months), category_spending breakdowns

transactions.json

Recent procurement and operational transactions

purchase_orders.json

Purchase orders created for backlog items. Initially empty — created at runtime via POST
